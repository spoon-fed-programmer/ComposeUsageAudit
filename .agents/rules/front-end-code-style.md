# 🚀 Web Development Antigravity Rules

> 소프트웨어가 비대해지면서 아래로 잡아당기는 인력(기술 부채, 복잡성)을 거스르고, 코드를 항상 가볍고 날렵하게 유지하기 위한 웹 프론트엔드 개발 원칙입니다. **모듈화**와 **스타일 일관성**을 핵심 가치로 삼습니다.

---

## 1. 컴포넌트 중력 제어 (Encapsulation Rule)
하나의 파일이나 컴포넌트가 너무 많은 책임을 가지면 복잡성의 중력이 작용해 코드가 무거워집니다. 모든 컴포넌트는 단일 책임 원칙을 따릅니다.

* **UI와 비즈니스 로직의 철저한 분리**
  * 화면을 그리는 UI 컴포넌트(Presentational)에는 복잡한 비즈니스 로직, 데이터 페칭/파싱, 상태 관리 코드를 가급적 직접 포함하지 않습니다.
  * 상태와 비즈니스 로직은 **Custom Hooks(또는 전용 컨텍스트/유틸리티)**로 분리하여 격리합니다.
* **자기 완결성 (Self-Containment)**
  * 하나의 컴포넌트는 그 자체로 독립적인 모듈이어야 합니다. 
  * "컴포넌트 폴더/파일을 복사해서 다른 화면이나 기능으로 옮겨도 최소한의 수정으로 작동해야 한다"는 기준을 지킵니다.
* **책임 분리 및 모듈화 강제 (Separation of Concerns & Low Depth)**
  * 코드 생성을 완료할 때는 반드시 함수나 모듈의 책임을 명확히 나누고 모듈화를 진행합니다.
  * 개별 메서드 및 함수의 코드 중첩(nesting) depth가 깊어지지 않도록(예: 최대 2 depth 이내 권장) 보조 함수나 유틸리티로 쪼개어 가독성과 유지보수성을 극대화합니다.

## 2. 스타일의 무중력 상태 (Zero Global CSS Rule)
스타일 일관성을 유지하려다 전역 스타일 파일이 비대해지는 현상을 방지하고, 디자인 시스템을 강제합니다.

* **임의의 하드코딩 값 금지 (Magic Value Zero)**
  * `margin: 13px;`, `color: #ff33aa;`와 같이 임의로 정의된 매직 넘버/컬러 사용을 전면 금지합니다.
  * 모든 스타일은 사전에 정의된 **디자인 토큰(Color, Spacing, Typography, Radius 등 - Tailwind Config에 정의됨)**의 변수만 사용하여 일관성을 유지합니다.
* **컴포넌트 스코프 스타일링 지향**
  * 전역 CSS 오염을 막기 위해 **Tailwind CSS(Utility-first)**를 기본으로 활용합니다.
  * 스타일이 컴포넌트 모듈 내부에 종속되거나 유틸리티화되어, 페이지가 늘어나도 전체 스타일 파일의 크기가 일정하게 유지되도록 합니다.

---

## 3. 기능 및 도메인 기반 응집 (Colocation Rule)
관련된 코드들이 프로젝트 곳곳에 멀리 떨어져 있으면 탐색 비용이 늘어나고 모듈화가 깨집니다.

* **기능 중심의 폴더 구조 (Feature-Driven Structure)**
  * `components/`, `hooks/`, `utils/` 등의 분류를 유지하되, 특정 화면이나 탭 전용으로 사용되는 컴포넌트는 하위 디렉토리(예: `components/FilesTab/`)로 묶어 응집도를 높입니다.
  * *Bad*: 모든 화면과 탭의 자식 컴포넌트들이 구분 없이 하나의 거대한 `components/` 폴더에 뒤섞여 있는 구조
  * *Good*: 특정 기능을 하나의 독립된 패키지처럼 관리하는 구조
    ```text
    viewer/src/
    ├── components/
    │   ├── FilesTab/
    │   │   ├── FileDetailPanel.jsx    # UI 컴포넌트
    │   │   └── FileMenuList.jsx       # UI 컴포넌트
    │   └── TabSwitcher.jsx            # 공통/레이아웃 컴포넌트
    ├── hooks/
    │   └── useFileCsv.js              # 비즈니스 로직 (Hook)
    └── utils/
        └── csvParser.js               # 파일 파서 유틸리티
    ```
* **손쉬운 제거 (Easy Deletion)**
  * 특정 기능이나 탭이 더 이상 필요 없어졌을 때, 해당 기능의 **폴더나 파일만 삭제하면 프로젝트 전체에서 사이드 이펙트 없이 깔끔하게 제거**될 수 있도록 결합도를 낮춥니다.

## 4. 의존성 역전 규칙 (Loose Coupling Rule)
외부 라이브러리나 특정 데이터 포맷에 컴포넌트가 강하게 결합되면, 코드가 무거워져 리팩토링이나 마이그레이션이 불가능해집니다.

* **Props 중심의 인터페이스 설계**
  * 컴포넌트는 내부의 구체적인 구현이 아니라, 외부에서 주입받는 인터페이스(Props)에만 의존해야 합니다. 내부 구현이 바뀌어도 인터페이스가 유지된다면 다른 모듈에 영향을 주지 않습니다.
* **외부 라이브러리 및 데이터 파서 격리 (Adapter Pattern)**
  * CSV 파서나 특정 데이터 포맷 분석 라이브러리를 컴포넌트 내부에서 직접 결합하여 사용하지 않습니다.
  * `utils/csvParser.js`와 같은 별도 모듈로 파싱 기능을 캡슐화한 뒤, 컴포넌트는 정제된 데이터 객체만 주입받아 사용하도록 격리시킵니다.

---

## 5. TDD 기반 비대화 방지 (Test-Driven Antigravity)
구현 코드보다 테스트 케이스(TC)를 먼저 작성하는 TDD(Test-Driven Development)는 코드의 과설계를 막고 모듈화를 강제하는 강력한 억제기입니다.

* **Red-Green-Refactor 사이클 준수**
  * **Red (실패):** 구현 코드를 작성하기 전, 요구사항을 만족하는 실패하는 테스트 케이스를 먼저 작성합니다.
  * **Green (성공):** 테스트를 통과할 만큼의 '최소한의 오버헤드가 없는 실제 코드'만 구현합니다. 기능이 늘어나는 중력을 제어합니다.
  * **Refactor (리팩토링):** 스타일 일관성과 모듈화 규칙을 적용하며 코드를 정제합니다. 이미 견고한 테스트 코드가 있으므로 안전한 구조 개선이 가능합니다.
* **테스트가 불가능한 코드는 설계 오류**
  * 특정 모듈이나 Hook이 테스트 케이스를 작성하기 너무 복잡하다면, 그것은 모듈화가 실패했거나 너무 많은 중력(책임)을 가지고 있다는 강력한 신호입니다. 즉시 컴포넌트를 더 작은 단위로 쪼개야 합니다.

| **정적 분석 규칙** | **Oxlint** | 빠른 정적 린팅을 통해 일관된 코드 품질 유지 |
